# 2. hive表体系和pipeline优化设计

## 2.1 新解析pdf任务包含已解析pdf的情况

现在需要考虑如下情况：
已经解析了一部分pdf，业务方新提交的一批pdf任务中(只有pdf的url)，但其实有部分pdf已经解析过了，此时需要跳过已经解析过的pdf，只解析未解析过的pdf。并且在给业务方返回结果时，需要包含已经解析过的pdf整体交付。

根据这个需求帮我优化ocr的表结构设计，并实现对应的pipeline。

### 2.1.1 设计目标

- request 侧只提交 pdf\_url，但系统仍然能够识别哪些文档已经解析过
- pipeline 对已解析文档直接复用历史聚合结果，不再重复 render/OCR/aggregate
- 对业务方交付时，返回本次 request 的全量结果，包含复用文档和新解析文档
- 整个链路支持多次重复执行，表数据保持幂等

### 2.1.2 表结构设计

#### ODS: `ods_pdf_request_item`

每次业务方提交 request 时，先把本次提交快照落到 ODS。

字段：

- request\_id: 请求 ID
- owner: 业务方标识
- pdf\_url: 原始 PDF 地址
- doc\_id: 基于 PDF 文件 bytes 计算得到的 MD5
- file\_name: 文件名
- submit\_time: 提交时间

作用：

- 保留 request 原始输入快照
- 为后续任务编排提供 request 维度的输入基表

#### DIM: `dim_pdf_document`

一行表示一个逻辑文档，是全局去重后的文档主数据。

字段：

- doc\_id: 文档主键
- pdf\_url: 当前文档地址
- file\_name: 文件名
- file\_size\_bytes: 文件大小
- first\_seen\_time: 首次发现时间

作用：

- 只存储文档事实字段，不混入 page\_cnt、doc\_text 这类解析结果
- 作为全局文档主数据表

#### DWD: `dwd_doc_parse_status`

文档解析状态表，和文档事实表拆开维护。

字段：

- doc\_id
- parse\_status: `pending` / `parsed`
- parsed\_time
- last\_request\_id
- last\_seen\_time
- last\_update\_time

作用：

- 单独维护文档的解析状态
- 避免将状态量混入文档主数据表

#### DWD: `dwd_request_doc_task`

把 request 中的每个文档转换成可执行任务，并标记是否需要新解析。

字段：

- request\_id
- owner
- pdf\_url
- doc\_id
- file\_name
- submit\_time
- parse\_needed: 是否需要新解析

作用：

- 将 request 和解析任务解耦
- 明确哪些文档进入 render/OCR 主链，哪些文档直接走复用

#### DWD: `dwd_page_render`

页面渲染结果表，只记录本次真正需要解析的文档页面。

字段：

- doc\_id
- pdf\_url
- page\_id
- page\_no
- image\_uri
- render\_version
- process\_time
- source\_request\_id

#### DWD: `dwd_page_ocr`

页面 OCR 结果表，只处理新增页面。

字段：

- doc\_id
- page\_id
- ocr\_text
- ocr\_confidence
- ocr\_version
- process\_time
- source\_request\_id

#### DWD: `dwd_doc_aggregate`

文档聚合结果表，作为可复用的标准交付底座。

字段：

- doc\_id
- pdf\_url
- page\_cnt
- doc\_text
- quality\_score
- aggregate\_version
- process\_time
- source\_request\_id

作用：

- 存储文档级最终 OCR 结果
- 后续 request 命中同一 doc\_id 时可直接复用

#### ADS: `ads_request_delivery_item`

业务方交付事实表，一次 request 一份全量结果。

字段：

- request\_id
- owner
- doc\_id
- pdf\_url
- delivery\_source: `cached` / `newly_parsed`
- page\_cnt
- doc\_text
- quality\_score
- delivery\_time

作用：

- 对业务方按 request 交付全量结果
- 清晰区分哪些结果来自缓存复用，哪些来自本次新解析
- 只保留交付事实字段，不混入状态量

#### ADS: `ads_request_delivery_status`

业务方交付状态表。

字段：

- request\_id
- doc\_id
- delivery\_status: `delivered`
- status\_time

作用：

- 单独维护交付状态
- 与交付事实表分开，避免状态字段污染事实表

### 2.1.3 pipeline 改造

新的 pipeline 顺序如下：

1. `step0_ingest_request.py`

- 扫描 `data/object_storage/raw_pdfs` 中的 PDF，生成一个 request 快照
- 对每个 PDF 文件读取 bytes 并计算 MD5，作为 `doc_id`
- 更新 `dim_pdf_document`
- 读取 `dwd_doc_parse_status` / `dwd_doc_aggregate` 判断哪些 doc\_id 已有历史结果
- 生成 `dwd_request_doc_task`，标记 `parse_needed`
- 更新 `dwd_doc_parse_status`

1. `step1_render.py`

- 只处理 `parse_needed = true` 的文档
- 对已存在渲染结果的 doc\_id 跳过，保证幂等

1. `step2_ocr.py`

- 只处理本次新增页面
- 对已有 OCR 结果的 page\_id 跳过

1. `step3_aggregate.py`

- 只聚合本次新解析文档
- 写回 `dwd_doc_aggregate`
- 更新 `dwd_doc_parse_status.parse_status = parsed`

1. `step4_validation.py`

- 按 request 将 `dwd_request_doc_task` 与 `dwd_doc_aggregate` 关联
- 产出 `ads_request_delivery_item`
- 单独产出 `ads_request_delivery_status`
- 返回本次 request 的全量交付结果

### 2.1.4 命中已解析 PDF 时的执行逻辑

当业务方发起一个新 request，且其中部分 PDF 已经解析过时：

1. 先依据 `pdf_url -> doc_id` 做文档归一
2. `doc_id` 直接使用 PDF 文件 bytes 的 MD5，因此同内容文件天然归并到同一个文档
3. 用 `doc_id` 去命中 `dwd_doc_aggregate`
4. 命中的文档标记为 `reused`，不进入 render/OCR/aggregate
5. 未命中的文档标记为 `pending_parse`，进入解析主链
6. 在交付阶段统一从 `dwd_doc_aggregate` 取结果，形成 request 全量输出

这样就能同时满足：

- 新文档继续增量解析
- 老文档复用历史结果
- 交付结果仍然按 request 全量返回

### 2.1.5 demo 中的实现约束

- 当前 demo 以本地文件路径模拟 `pdf_url`
- `doc_id` 固定使用 PDF 文件 bytes 的 MD5，不再依赖文件名或 URL
- 由于 demo 不拉取远端文件，所以 render 步骤当前仅支持本地路径或 `file://` 路径
- 当前目录按存储方式拆分为两类：`data/object_storage` 模拟对象存储，`data/hive_warehouse` 模拟 Hive 表存储

每个业务方提出的需求是一个request，
